# 会话式 TBM Supervisor Agent 设计说明

## 1. 文档目的

这份文档用于说明当前仓库中“智能问答”部分的**最新版实现**。  
当前系统已经不再保留旧版单体 `TBMAgent` 入口，只保留一个实际在用的问答体系：

- `POST /api/tbm/agent_v2`
- `GET /api/tbm/agent_v2/capabilities`
- `GET /api/tbm/agent_v2/session`

它的定位不是开放域聊天机器人，而是：

**一个面向 TBM 工程分析问答的、带会话记忆的 supervisor-style agent。**

---

## 2. 当前结构

当前问答链路可以概括为 5 层：

```text
用户问题
  -> 前端抽屉式问答界面
  -> FastAPI 问答路由
  -> TBMSupervisorAgent
  -> Domain Agents
  -> TBMTools
  -> 原有 TBM 分析服务
  -> 结构化回答 / 调度计划 / 工具执行摘要 / 会话记忆
```

它不是传统的“一个大模型直接看数据回答”，而是：

- `Supervisor` 负责规划
- `Domain Agent` 负责领域路由
- `TBMTools` 负责真实工具执行
- `SQLite` 负责保存会话和消息
- `LLM` 只在用户开启“LLM 润色”时参与最终表述优化

---

## 3. 为什么只保留这一套

旧版问答入口原来是一个单体规则 agent，它可以工作，但有几个明显问题：

- 没有真正的会话记忆
- 前后端实际已经默认改用新版 supervisor 流程
- 保留两套入口会导致接口、文档和维护逻辑重复

因此现在仓库中只保留一套最新版问答体系：

**会话式 Supervisor Agent**

这样可以让：

- 前端只维护一套交互
- 后端只维护一套问答入口
- 文档、测试、接口表保持一致

---

## 4. 后端架构

### 4.1 路由层

问答入口在 `backend/routes/tbm.py` 中，当前保留 3 个相关接口：

- `POST /api/tbm/agent_v2`
- `GET /api/tbm/agent_v2/capabilities`
- `GET /api/tbm/agent_v2/session`

其中：

- `agent_v2` 负责运行一次问答分析
- `capabilities` 返回当前可用专家和工具
- `session` 返回指定会话的历史消息，用于前端恢复上下文

### 4.2 Supervisor 层

当前核心类是：

- `TBMSupervisorAgent`

它承担以下职责：

- 接收用户 query
- 结合 `session_id` 读取历史上下文
- 从当前问题中提取日期、追问意图和主题
- 生成调度计划
- 调用对应的领域 agent
- 汇总答案、关键指标、调度计划和工具执行结果
- 将本轮对话写入会话存储

### 4.3 Domain Agent 层

系统内部按专业领域拆成 6 个角色：

| Agent | 作用 |
| --- | --- |
| `DataAgent` | 日期、CSV 读取、综合日分析 |
| `OperationAgent` | 工况、停机、效率、施工状态 |
| `SafetyAgent` | 气体统计与超限 |
| `GeologyAgent` | 地质融合、前方风险、耦合、剖面 |
| `TwinAgent` | 数字孪生状态快照 |
| `MemoryAgent` | 历史对比 |

这些 agent 不是 6 个独立大模型，而是 6 个**领域角色包装器**。  
真正执行分析的是下一层的 `TBMTools`。

### 4.4 工具层

问答系统复用既有分析链路，并将其封装为统一工具层。  
当前主要工具包括：

- `list_dates`
- `load_day`
- `analyze_day`
- `analyze_operation`
- `analyze_gas`
- `analyze_geology`
- `analyze_forward_risk`
- `get_digital_twin_state`
- `compare_history`
- `risk_profile`

这样做的意义是：

- 问答层不直接操作原始分析代码
- 所有工具调用结果都有统一结构
- 前端可以展示“工具执行轨迹”

---

## 5. 会话记忆是怎么做的

当前版本最大的升级之一，是增加了**持久化会话记忆**。

### 5.1 存储方式

会话和消息保存在 `SQLite` 中，主要有两张表：

- `agent_sessions`
- `agent_messages`

`agent_sessions` 存放会话级摘要，例如：

- `session_id`
- `title`
- 最近使用日期
- 上一轮问题
- 上一轮关键 highlights

`agent_messages` 存放逐条消息，例如：

- 用户问题
- 助手回答
- 结构化 payload
- 时间戳

### 5.2 会话恢复

前端每次打开问答抽屉时，会读取本地保存的 `session_id`，然后调用：

- `GET /api/tbm/agent_v2/session`

这样即使页面刷新，也能恢复之前的问答记录。

### 5.3 会话里保留了什么上下文

当前系统会重点继承这些内容：

- 上一轮日期
- 上一轮用户问题
- 上一轮路由到的专家
- 上一轮重点区段
- 上一轮前方风险等级

这使得系统能处理如下追问：

- “为什么那一段值得注意”
- “继续展开说说”
- “和上次比有什么变化”
- “那前方还要注意什么”

## 6. 当前规划方式

当前版本不是纯 ReAct，而是：

**规则规划 + 会话上下文增强**

### 6.1 基础规划

Supervisor 会先根据关键词识别问题主题，例如：

- “工况、停机、效率” -> `OperationAgent`
- “气体、瓦斯、超限” -> `SafetyAgent`
- “地质、围岩、耦合” -> `GeologyAgent`
- “数字孪生、当前里程” -> `TwinAgent`
- “历史、对比、变化” -> `MemoryAgent`

### 6.2 上下文增强

如果当前问题看起来是追问，系统会自动继承上一轮上下文，例如：

- 继承上一轮日期
- 继承上一轮焦点区段
- 继承上一轮已命中的专家方向

这就让当前规划从“只看这一句话”变成“结合上一轮继续分析”。

### 6.3 为什么没有做成完全自由 ReAct

当前项目定位是工程分析问答，不是开放域助手。  
完全自由的 ReAct 虽然灵活，但会带来：

- 工具调用不稳定
- 响应时间增长
- 可重复性下降
- 解释链路更难控

所以现阶段采用的是更稳的混合模式：

- 简单问题：规则直接路由
- 追问问题：规则 + 上下文增强
- 文本润色：可选 LLM

## 7. 当前响应结构

`POST /api/tbm/agent_v2` 当前返回的核心字段包括：

- `session_id`
- `query`
- `effective_query`
- `date`
- `mode`
- `verbose`
- `supervisor_plan`
- `routed_agents`
- `answer`
- `highlights`
- `context_summary`
- `agent_results`
- `tool_results`

其中：

- `answer`：前端主展示文本
- `highlights`：摘要指标
- `supervisor_plan`：本轮调度计划
- `tool_results`：工具执行摘要
- `context_summary`：本轮使用了哪些会话上下文

这套结构兼顾了两件事：

- 普通用户可以直接读回答
- 技术人员可以展开看 agent 调度过程

---

## 8. 前端交互结构

前端当前已经不是原来的小浮窗，而是：

**右侧抽屉式问答界面**

### 8.1 当前交互特点

- 从仪表盘右下角按钮唤起
- 以右侧抽屉形式展开
- 打开时带遮罩
- 支持 `Esc` 关闭
- 中间是消息流滚动区
- 底部是固定输入区
- 支持“新对话”
- 支持恢复历史会话
- 支持折叠查看调度计划和工具执行结果

### 8.2 当前前端展示内容

每条助手回答会展示：

- 主回答文本
- 上下文摘要
- 关键指标卡片
- 预警/失败提示
- 调度计划
- 工具执行结果

这使得问答不只是“给一句话”，而是一个可以展开追踪的工程分析面板。

---

## 9. 当前版本的定位

当前这套问答系统最准确的定位是：

**一个带会话记忆的、上下文规划型、可追踪工具链路的 TBM Supervisor Agent。**

它已经具备：

- 问答入口
- 多专家角色
- 统一工具层
- 会话记忆
- 追问能力
- 可选 LLM 润色
- 前端可视化调度轨迹

但它仍然不是一个“完全自由思考的通用 agent”。  
它更适合被理解为：

**工程分析场景下的受控智能问答系统。**

---

## 10. 当前最值得继续演进的方向

如果后续继续升级，最值得推进的是：

1. 为复杂解释型问题加入有限步 ReAct 子循环  
   只在复杂追问时启用，不破坏当前稳定性。

2. 增加“回答目标模式”  
   例如“答辩模式”“现场值班模式”“报告摘要模式”。

3. 让会话记忆进一步结构化  
   例如显式记录最近一次重点区段、最近一次建议、最近一次异常来源。

4. 给 agent 问答补更多自动化测试  
   包括会话恢复、追问继承、失败兜底、LLM 润色模式。

---

## 11. 一句话总结

当前仓库中的最新版问答系统，不再是旧版单体 agent，而是：

**一个以 `agent_v2` 为唯一入口、带会话记忆、以 supervisor 为核心、通过领域工具完成 TBM 工程分析问答的最新实现。**
