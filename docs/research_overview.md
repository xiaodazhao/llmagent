# 数字孪生和大语言模型协同驱动的 TBM 施工报告自动化生成

## 1. 文档目的

这份文档用于帮助导师、师兄或项目协作者在不逐行阅读代码的情况下，快速理解本项目的研究目标、数据来源、处理流程、融合逻辑、核心指标和报告生成链路。

项目的核心目标不是重新发明一个地质风险预测器，而是围绕“施工报告自动化生成”这一任务，构建一个由数字孪生和大语言模型协同驱动的完整技术框架。

更准确地说，本项目解决的是这样一个问题：

- TBM 日常施工数据是高频时序数据，字段多、噪声大、直接阅读困难。
- 超前地质预报和掌子面素描以 PDF 形式存在，包含大量自然语言和表格信息，无法直接用于自动报告生成。
- 大语言模型如果直接面对原始 CSV 和 PDF，容易出现事实遗漏、工程语言不稳定、当前掌子面与前方预测混写、风险表述过度等问题。

因此，本项目采取的基本思路是：

1. 从 TBM 日运行数据和已有地质预报资料中提取结构化信息。
2. 将多源数据统一到同一个时间-里程框架中。
3. 构建轻量化施工数字孪生状态。
4. 将孪生状态转换为受控提示词，驱动大语言模型生成正式施工报告。

## 2. 项目定位

从研究角度看，这个项目最合适的表述不是“风险识别系统”，而是：

**一个面向 TBM 施工报告自动化生成的数字孪生-大语言模型协同框架。**

这里有两个关键界定。

第一，地质风险信息并不是系统从零预测得到的。  
项目使用的是外部已有的地质资料，包括：

- `TSP` 超前地质预报
- `HSP` 超前地质预报
- `掌子面/洞身素描`

这些资料本身已经包含专家或设备给出的地质结论。系统的任务是将这些结论结构化、标准化、里程化，并与 TBM 运行数据融合。

第二，大语言模型并不直接读取原始工程文件。  
系统先把原始数据加工为一个轻量化数字孪生状态，再由 LLM 基于这个孪生状态自动生成报告。也就是说，LLM 承担的是“工程化写作”角色，而数字孪生承担的是“状态组织与事实约束”角色。

## 3. 整体技术路线

项目整体可以概括为四层：

1. 数据输入层  
   TBM 日运行 CSV、超前地质预报 PDF、掌子面素描 PDF。

2. 结构化处理层  
   对 CSV 做时间清洗、工况分段、状态识别、气体统计；对 PDF 做文本提取、规则解析和证据标准化。

3. 数字孪生建模层  
   以统一里程轴为桥梁，把地质证据挂接到 TBM 数据上，形成逐里程和区段级的施工数字孪生状态。

4. LLM 协同生成层  
   将结构化孪生状态转写为受控提示词，驱动大语言模型自动生成日报和时段分析报告。

如果用一句话总结：

**项目不是让模型直接“看原始数据写报告”，而是先“构建状态”，再“基于状态写报告”。**

## 4. 原始数据来源与使用方式

### 4.1 TBM 日运行 CSV

TBM 日运行 CSV 是系统的主数据源，反映当天施工设备和运行过程的实际表现。

项目中主要使用的字段包括：

- 时间字段：
  - `运行时间-time`

- 位置字段：
  - `chainage`
  - 若缺少统一里程字段，则使用 `导向盾首里程` 或 `开累进尺` 转换得到

- 施工状态字段：
  - `掘进状态`

- 运行参数字段：
  - `推力`
  - `推进速度`
  - `推进给定速度`
  - `刀盘扭矩`
  - `刀盘实际转速` 或 `刀盘转速`
  - `贯入度`

- 气体监测字段：
  - `CO2检测`
  - `H2S检测`
  - `SO2检测`
  - `NO2检测`
  - `NO检测`
  - `CH4检测`

这些字段共同描述的是“当天施工过程中的运行状态、效率表现、工况变化和安全监测情况”。

### 4.2 地质证据 PDF

项目接入的地质资料主要有三类。

#### 4.2.1 TSP 超前地质预报

主要提取：

- 报告编号
- 报告日期
- 预报范围起止里程
- 当前掌子面里程
- 下次预报里程
- 结论段文字
- 表格中各区段的岩性、围岩等级、风险描述和关键物性参数

#### 4.2.2 HSP 超前地质预报

主要提取：

- 预报范围起止里程
- 反射异常强度
- 掉块相关位置提示
- 围岩等级建议
- 节理裂隙发育情况
- 岩体破碎程度
- 风化程度
- 稳定性描述

#### 4.2.3 掌子面/洞身素描

主要提取：

- 当前掌子面里程
- 围岩级别建议
- 风化程度
- 裂隙发育情况
- 岩体破碎程度
- 出水信息
- 掉块信息
- 自稳性
- 岩性描述

这类资料和 TSP/HSP 的最大差别在于，它通常反映的是“当前掌子面现场揭示情况”，而不是“前方区段预测结果”。

## 5. CSV 数据处理逻辑

### 5.1 时间清洗与排序

系统首先将 `运行时间-time` 转换为统一时间格式，删除无效记录，并按时间排序。  
这一步保证后续所有分析都是在连续时间序列上完成的。

### 5.2 统一里程轴

如果原始数据已经包含 `chainage` 字段，则直接使用。  
如果没有，则优先从 `导向盾首里程` 或 `开累进尺` 中构造统一里程。

统一里程轴是整个系统的关键，因为后续所有地质资料都需要按里程挂接到 TBM 运行数据上。

### 5.3 基础工况分段

项目先将 TBM 运行过程划分为 4 类基础工况：

- 停机
- 启动/过渡
- 稳定掘进
- 异常扭矩

判定逻辑主要基于：

- `掘进状态`
- `推力`
- `推进速度`
- `刀盘扭矩`

基本思想是：

- `掘进状态 = 0` 往往表示停机；
- 推力和推进速度同时有效时，倾向于判为稳定掘进；
- 推力存在但推进速度接近 0 时，判为启动或过渡；
- 扭矩存在但推进没有形成时，视为异常扭矩状态。

完成分类后，系统会把连续同状态样本合并成工况段，并统计：

- 各类工况段数量
- 各类工况累计时长
- 最长工况段
- 是否存在大量短时启停或碎片化切换

这一步的作用，是为日报中的“总体运行情况”“基础工况统计分析”提供基础事实。

### 5.4 隐含施工状态识别

除了基础工况分类之外，项目还在“工作中”的样本上做了一层更细粒度的施工状态识别。

使用的输入字段主要是：

- `推力`
- `刀盘扭矩`
- `刀盘实际转速`
- `推进速度`

方法上采用无监督聚类，而不是监督分类。  
聚类结果再根据各状态的平均特征进行语义解释，例如：

- 高负载稳定推进状态
- 高推力低速受阻状态
- 低负载快速推进状态
- 低负载调整状态

随后系统会继续统计：

- 各状态出现的时段
- 各状态占比
- 状态切换频率
- 不同状态下的推进效率、推力、扭矩等特征

这一步的作用是让报告能从“工况段统计”进一步深入到“施工状态与效率分析”。

### 5.5 气体监测分析

如果 CSV 中存在气体字段，系统会分别对：

- 全天
- 掘进期
- 停机期
- 可选的施工状态期

进行统计分析。  
输出内容包括：

- 平均值
- 最大值
- 最小值
- 超阈值事件次数
- 超阈值事件持续时间

因此，报告中不仅会体现设备与工况，还会纳入安全监测维度。

## 6. PDF 证据处理逻辑

项目对 PDF 的处理不是简单存档，而是做“文本提取 + 规则解析 + 结构化标准化”。

### 6.1 文本提取与规范化

系统先从 PDF 中提取全文文本，再统一：

- 全角和半角符号
- 换行
- 波浪线和破折号
- 空白字符

这样做是为了让后续正则解析在不同格式的 PDF 上更稳定。

### 6.2 按文档类型分别解析

对于 `TSP`、`HSP` 和 `素描`，系统采用不同的规则集分别解析。  
因为三类文档的结构不同、关注字段不同、表达习惯也不同。

### 6.3 统一结构化证据格式

所有解析出来的信息最终都会统一为标准证据记录。  
每条证据至少包含：

- `source_type`：证据来源类型
- `report_id`：报告编号
- `start_num`：起始里程
- `end_num`：终止里程
- `source_level`：证据层级
- `raw_text`：原始文本
- `attrs_json`：结构化属性

其中 `attrs_json` 可能包含：

- 围岩等级
- 风险等级
- 出水标记
- 掉块标记
- 变形标记
- 节理裂隙情况
- 岩体破碎情况
- 稳定性
- 岩性
- 风险标签

### 6.4 证据导入与去重

导入时系统会做：

- PDF 路径展开
- 来源类型识别
- 证据格式清洗
- `evidence_id` 去重
- 区段长度合理性约束
- 支持“仅预检”和“正式入库”两种模式

最终证据既保存在 CSV 中，也同步到 `SQLite`，方便后续增量更新和快速读取。

## 7. 数字孪生的核心：按里程进行地质-施工融合

这一层是整个项目的核心，也是“数字孪生”真正成立的地方。

项目的关键问题是：

**TBM 数据是按时间记录的，而地质资料是按里程描述的，必须先把它们统一到同一个空间框架中。**

### 7.1 证据分层

融合前，系统会区分不同证据层级：

- `segment`：区段型证据
- `report_conclusion`：报告结论型区段证据
- `point`：单点型证据，例如掌子面素描
- `overview`：若仅为整份报告概括而无明确空间约束，则不参与逐里程融合

### 7.2 对每个里程点寻找活跃证据

对于 TBM 数据中的每一个有效里程点，系统都会在证据库中寻找命中的地质证据。

命中规则为：

- 对于区段型证据，只要该里程点位于证据作用区间内，视为命中；
- 对于点位型证据，则在点位附近给一个缓冲范围，只要进入缓冲区间，也视为命中；
- 为避免边界过于僵硬，还会使用一定的空间容差。

最终，每个里程点会得到一个“当前命中的地质证据集合”。

### 7.3 融合输出的不是风险判决，而是事实型地质状态

对于命中的证据集合，系统先做“事实融合”，输出一组结构化地质状态变量，而不是直接输出“灾害已发生”之类的强结论。

融合后的核心字段包括：

- `fused_grade`：综合围岩等级
- `hazard`：综合灾害标签
- `water_flag_fused`：是否存在出水关注
- `collapse_flag_fused`：是否存在掉块关注
- `deformation_flag_fused`：是否存在变形关注
- `joint_degree_fused`：裂隙发育程度
- `rock_mass_state_fused`：岩体破碎程度
- `weathering_fused`：风化程度
- `stability_fused`：稳定性
- `lithology_fused`：岩性
- `active_source_count`：独立有效来源数
- `weighted_evidence_strength`：证据强度
- `uncertainty`：证据不确定性

### 7.4 具体融合逻辑

#### 围岩等级

不同来源对围岩等级可能存在差异。系统采用偏保守的融合逻辑，优先保留更差等级，而不是简单平均。

#### 灾害标签

如果任一来源提到：

- 出水
- 掉块
- 变形
- 裂隙密集
- 围岩破碎

这些标签都会进入综合标签集合，避免遗漏潜在关注点。

#### 二值标记

对于 `water_flag`、`collapse_flag`、`deformation_flag` 这类二值字段，通常采用逻辑“或”融合：只要任一证据来源明确提示，就认为当前里程点具有该关注。

#### 来源数

系统不会简单统计记录条数，而是对 `source_type + report_id` 做去重后再统计独立来源数。  
这样可以避免同一份报告中的重复片段把“多源支持”夸大。

#### 不确定性

不确定性不是统计置信区间，而是证据覆盖意义上的不确定性。  
来源越多，不确定性越低；来源越少，不确定性越高。

#### 证据强度

系统会根据证据层级、置信度和一致性，对不同证据赋予权重，并求和得到 `weighted_evidence_strength`。  
这不是风险分数本身，而是“证据支撑力度”的量化表达。

## 8. 区段化处理：从逐点状态到报告对象

逐里程点状态虽然细，但不适合直接写报告，因此系统进一步按固定里程长度进行区段化。当前默认是每 `10m` 一个区段。

区段化以后，每个区段会形成一组更适合报告组织的特征：

### 8.1 施工响应特征

- 平均推进速度
- 推进速度波动
- 平均推力
- 推力波动
- 平均刀盘扭矩
- 扭矩波动
- 平均转速
- 平均贯入度
- 推进效率
- 近似停机比例
- 近似零速比例

### 8.2 地质状态特征

- 区段平均地质关注度
- 区段最大地质关注度
- 主导围岩等级
- 主导灾害标签
- 活跃来源数
- 证据不确定性

区段是后续 3 个核心指标和报告重点排序的基本对象。

## 9. 三个核心指标的定义与作用

这三个指标的设计目标，不是“替代专家做地质风险识别”，而是：

**把已有地质证据和施工响应压缩成报告可用、可排序、可解释的区段级状态指标。**

### 9.1 GRS：Geology Risk Score

在本项目中，更准确地说，GRS 表示的是：

**区段地质关注度摘要指标。**

它不是严格意义上的地质风险真值，也不是灾害发生概率，而是把多源地质证据压缩为一个区段级关注度分数。

#### 9.1.1 逐点基础分数

对每个里程点，系统先计算一个基础分数：

`GRS_row_base = 0.30*grade_score + 0.25*hazard_score + 0.20*water_score + 0.15*collapse_score + 0.10*source_confidence`

其中：

- `grade_score`：围岩等级映射分数  
  围岩等级越差，分数越高。大致映射为：
  - I -> 0.05
  - II -> 0.20
  - III -> 0.40
  - IV -> 0.70
  - V -> 1.00

- `hazard_score`：灾害关键词分数  
  根据 `hazard / risk / active_sources` 等文本中出现的关键词累加，如断层、破碎、裂隙、异常反射、软弱、岩溶等，不同关键词有不同权重。

- `water_score`：出水关注分数  
  若存在明确 `water_flag` 或文本中出现出水、涌水、突水、富水等表达，则该项提高。

- `collapse_score`：掉块/坍塌关注分数  
  若存在明确 `collapse_flag` 或文本中出现掉块、坍塌等表达，则该项提高。

- `source_confidence`：来源支撑强度  
  如果有 `active_source_count`，则按来源数进行归一化；如果只有覆盖类型或来源文本，则用单源/多源信息进行粗略映射。

#### 9.1.2 区段汇总

对于每个区段内的所有逐点分数，系统统计：

- `GRS_mean`
- `GRS_max`

并定义区段基础分数：

`GRS_base = 0.35*GRS_mean + 0.65*GRS_max`

这里让最大值占更高权重，是因为报告更关心区段内是否存在明显高关注位置，而不是仅看平均水平。

#### 9.1.3 动态修正

项目没有将 GRS 固定为纯地质先验，而是引入施工响应进行动态修正：

`GRS_corrected = GRS_base * (1 + λ*(0.5*RAI + 0.5*stop_ratio))`

其中：

- `λ = 0.40`
- `RAI`：施工响应异常指数
- `stop_ratio`：区段停机或近似停机程度

其含义是：

如果某段本来就具有较高地质关注度，同时又出现了明显施工异常，那么这段在报告中应被更强地关注。

#### 9.1.4 空间平滑

为避免单个点位噪声导致区段分数过于尖锐，系统对修正后的 GRS 做相邻区段平滑：

- 前一区段权重 0.25
- 当前区段权重 0.50
- 后一区段权重 0.25

#### 9.1.5 GRS 的实际作用

GRS 在本项目中的作用不是“宣布风险”，而是：

- 为报告提供“哪些区段值得重点关注”的排序依据；
- 将已有多源地质预报结论压缩成一个可解释的数值摘要；
- 为后续 GRCI 耦合判断提供地质侧输入。

### 9.2 RAI：Response Anomaly Index

RAI 表示的是：

**区段施工响应异常摘要指标。**

它不表达地质结论，只表达施工过程在某个区段是否存在异常表现。

#### 9.2.1 异常子分量

系统先计算多个区段级异常分量，主要包括：

- `speed_drop_score`：推进速度显著偏低程度
- `speed_volatility_score`：推进速度波动异常程度
- `thrust_anomaly_score`：推力异常程度
- `torque_anomaly_score`：扭矩异常程度
- `rpm_anomaly_score`：转速异常程度
- `penetration_anomaly_score`：贯入度异常程度
- `efficiency_anomaly_score`：推进效率异常程度

这些分数并不简单依赖固定阈值，而是基于当天全体区段的稳健统计特征进行归一化，衡量“某段是否显著偏离同日正常区段”。

#### 9.2.2 停机异常

系统单独构造 `stop_anomaly`，优先使用显式停机比例字段。  
若缺失，则退化为：

- 工作比例的反向值
- 区段内近似零速比例
- `掘进状态 = 0` 的比例

这一步主要捕捉“停顿或准停顿”现象。

#### 9.2.3 效率异常

`efficiency_anomaly` 优先通过工作比例或推进效率计算。  
如果这些字段缺失，则退化为基于速度下降的近似判断。

它主要表达的是：

**即便未完全停机，区段施工是否已经表现出效率衰减。**

#### 9.2.4 参数异常

系统还会综合多个参数的异常程度，构造 `param_anomaly`：

`param_anomaly = 0.60*top_mean + 0.25*mean_all + 0.15*response_consistency`

其中：

- `top_mean`：异常最明显的几个参数分量的平均值
- `mean_all`：所有参数异常分量的整体平均
- `response_consistency`：多参数是否共同支持“该段存在异常”

这样可以避免被单个小波动误导，更强调多参数一致异常。

#### 9.2.5 最终 RAI

最终：

`RAI = 0.50*stop_anomaly + 0.30*efficiency_anomaly + 0.20*param_anomaly`

因此，RAI 本质上是在回答：

**这个区段的施工表现是否异常，以及异常程度有多高。**

#### 9.2.6 RAI 的实际作用

RAI 在本项目中的作用是：

- 为报告中的“施工稳定性分析”“局部异常分析”提供量化支撑；
- 与 GRS 共同构成后续耦合分析的输入；
- 帮助系统筛选那些地质关注可能已在施工参数中体现出来的区段。

### 9.3 GRCI：Geology Response Coupling Index

GRCI 表示的是：

**区段地质关注与施工响应之间的一致性和耦合程度。**

它的作用不是预测灾害，而是帮助报告回答：

**哪些区段不仅地质上值得关注，而且施工上也出现了对应的响应，因此更应该被重点写入报告。**

#### 9.3.1 同步耦合

如果某个区段同时具有较高的 GRS 和较高的 RAI，且两者水平接近，则同步耦合值高。  
这表示该段“地质关注”和“施工异常”在同一段上同时出现。

#### 9.3.2 滞后耦合

有些情况下，地质关注先出现，而施工异常在后续区段或后续片段中才体现出来。  
系统通过比较当前区段与前一区段的 GRS 和 RAI 变化，构造滞后耦合项，表达“风险先到、响应后现”的情况。

#### 9.3.3 响应变化耦合

系统还会关注进入某一区段后，施工响应是否发生明显抬升。  
如果地质关注进入该区段的同时，施工参数异常快速增加，那么该段的响应变化耦合值会上升。

#### 9.3.4 多参数一致性修正

如果施工异常不是单一参数的孤立波动，而是多参数共同印证，则该耦合关系更可信。  
因此 `response_consistency` 会进入耦合指标计算。

#### 9.3.5 来源置信修正

如果该区段的地质关注来自多个独立来源，则最终耦合结论更可信。  
因此系统会使用来源支撑强度对最终耦合分数做置信修正。

#### 9.3.6 最终 GRCI 公式

最终定义为：

`GRCI = (0.40*sync + 0.25*lag + 0.25*response_change + 0.10*multi_param) * confidence`

其中：

- `sync`：同步耦合
- `lag`：滞后耦合
- `response_change`：响应变化耦合
- `multi_param`：多参数一致性
- `confidence`：来源支撑修正

因此，GRCI 的高值并不单纯意味着“风险更大”，而更准确地意味着：

**该区段的地质关注和施工响应之间存在较强、较可信的对应关系。**

#### 9.3.7 区段分类

系统还根据 `GRS` 和 `RAI` 是否超过阈值，将区段划分为四类：

- A 类：地质关注高，施工响应也高  
  表示典型的地质-施工耦合高关注区段。

- B 类：地质关注高，但施工响应不强  
  表示前方地质预报关注明显，但当日施工尚未表现出强响应。

- C 类：地质关注不高，但施工响应高  
  表示施工异常明显，但地质证据不足，需进一步复核。

- D 类：地质关注和施工响应都不高  
  表示相对稳定区段。

这个分类非常适合服务报告写作，因为不同类别自然对应不同的叙述策略。

## 10. 轻量化数字孪生状态的构建

项目中的“数字孪生”不是三维几何孪生，而是面向报告生成的轻量化状态数字孪生。

它同步表达的不是设备外形，而是施工过程的关键状态。

当前系统构建的孪生状态包括六个维度：

### 10.1 时间状态

- 当天起始时间
- 当天结束时间
- 持续时长
- 样本数量

### 10.2 位置状态

- 起始里程
- 结束里程
- 当前里程
- 推进长度

### 10.3 运行状态

- 主导工况
- 掘进时长
- 停机时长
- 异常段数量
- 状态切换次数

### 10.4 地质状态

- 当前掌子面附近围岩等级
- 当前灾害标签
- 高关注区段数量
- 多源关注区段数量

### 10.5 安全状态

- 哪些气体出现超限
- 超限类型数量

### 10.6 前方状态

- 前方若干米范围内是否存在关注区段
- 主要灾害类型
- 建议关注级别
- 地质-施工耦合概况

因此，这个孪生体的本质是：

**对现实施工过程的结构化同步映射。**

它的最大作用不是展示，而是：

- 为大语言模型提供统一、受约束的输入状态；
- 保证报告中“当前掌子面”“前方区段”“施工状态”“气体监测”来自同一套状态空间；
- 使报告生成建立在可追踪、可解释的中间表示之上。

## 11. 大语言模型协同生成报告的逻辑

这一层的关键点是：

**大模型不是自由发挥，而是在数字孪生状态约束下进行受控写作。**

### 11.1 输入到 LLM 的不是原始文件，而是结构化中间结果

在调用 LLM 之前，系统已经准备好一组中间摘要，包括：

- 基础工况分段文本
- 基础工况统计文本
- 隐含施工状态文本
- 状态效率分析文本
- 状态统计文本
- 当前掌子面地质文本
- 前方区段地质融合文本
- 区段耦合分析摘要
- 数字孪生状态摘要
- 气体统计文本
- 历史记忆对比摘要

这意味着 LLM 的输入已经不是“原始数据”，而是经过结构化压缩后的状态表达。

### 11.2 报告生成的协同机制

协同逻辑可以概括为：

- 数字孪生负责“事实组织”
- 大语言模型负责“语言生成”

数字孪生先把异构数据转化为：

- 时间有序
- 空间对齐
- 层次分明
- 风险表达受约束

的状态摘要。

然后 LLM 再在这一基础上，完成：

- 工程报告语体组织
- 章节结构补全
- 历史变化表述
- 建议部分生成

### 11.3 日报生成逻辑

日报生成的流程是：

1. 读取某一天的 TBM CSV。
2. 完成工况分段、状态识别、地质融合、区段分析、数字孪生建模。
3. 生成当日结构化摘要。
4. 读取历史记录，构建“与前期相比”的变化摘要。
5. 将所有摘要写入受控提示词。
6. 调用大语言模型生成《TBM 综合施工工况分析报告》。

在提示词中，系统会明确要求模型：

- 不能把“关注”写成“已发生”
- 不能把掌子面现状与前方预测混写
- 不能照搬程序提示或原始字段名
- 不能虚构输入中不存在的信息
- 必须按固定章节生成正式工程报告

### 11.4 时段报告生成逻辑

时段报告与日报类似，但输入对象变成某个时间窗：

1. 先读取当天 CSV。
2. 再按用户指定的起止时间切出一个局部时间段。
3. 对这个时间窗重新完成同样的分析流程。
4. 生成更强调局部异常和局部波动的提示词。
5. 输出《TBM 施工时段工况分析报告》。

因此，日报解决的是“全天总结”，时段报告解决的是“局部复盘”。

### 11.5 LLM 配置策略

当前系统支持通过配置切换不同大模型提供方，已接入：

- DeepSeek
- Gemini

模型调用策略偏工程化和保守化：

- 较低 `temperature`
- 较高输出上限
- 明确错误返回
- 不做自由风格聊天，而是做正式报告输出

这说明项目对 LLM 的定位不是开放式问答，而是“受控文本生成器”。

## 12. 当前项目最合适的研究叙事

从研究角度看，项目最合适的表述不是：

**“我做了一个 TBM 风险识别系统。”**

而是：

**“我提出了一个由数字孪生和大语言模型协同驱动的 TBM 施工报告自动化生成框架。”**

更具体地说：

- 地质风险信息来自已有的超前地质预报和掌子面揭示资料；
- 施工状态信息来自 TBM 日运行 CSV；
- 项目的创新不在于替代专家重新识别风险，而在于：
  - 将异构工程数据统一到时间-里程状态空间中；
  - 构建轻量化施工数字孪生状态；
  - 设计区段地质关注、施工响应异常和耦合关注指标；
  - 在此基础上驱动大语言模型自动生成工程报告。

因此，如果用一句话概括研究主线：

**这个项目研究的是如何把多源 TBM 施工数据转化为一个可驱动大语言模型写作的数字孪生状态表示。**

## 13. 当前阶段的优势与不足

### 13.1 当前优势

- 已经打通从原始数据到报告生成的完整链路；
- 数据来源真实，场景明确，不是纯演示系统；
- 地质证据、施工数据和安全监测已经实现统一组织；
- 已形成结构化孪生状态，而不是仅靠模板拼接；
- 报告生成受到明确工程约束，不是完全自由生成；
- 具备一定工程化基础，包括缓存、SQLite、接口规范和测试。

### 13.2 当前不足

- 地质证据解析准确率还缺正式评估；
- 3 个指标目前更像“报告辅助指标”，缺少严格外部真值验证；
- 数字孪生的有效性还缺正式实验设计；
- 报告生成质量还缺人工对照评价；
- 当前的前方风险提示更接近“前视证据告警”，而不是严格预测模型。

## 14. 适合进一步请导师或师兄帮看的问题

- 论文标题下，是否应进一步强调“轻量化状态数字孪生”，避免被理解为三维几何孪生？
- `GRS / RAI / GRCI` 在论文中更适合称为“风险指标”还是“关注度/表征指标”？
- 报告生成效果应该采用什么评价体系最合适？
- 数字孪生的实验设计应如何体现“比直接用 LLM 更好”？
- 整个项目更适合写成“系统论文”还是“方法 + 系统应用”？

## 15. 一句话总结

本项目的真正贡献不是让大语言模型直接读取原始工程数据，而是：

**先将 TBM 施工过程中的多源异构数据构建为一个统一、可解释、可约束的轻量化数字孪生状态，再利用大语言模型基于该状态自动生成正式施工报告。**
