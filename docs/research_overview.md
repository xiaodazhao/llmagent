# 数字孪生和大语言模型协同驱动的 TBM 施工报告自动化生成

## 1. 文档目的

这份文档用于让协作者在不逐行阅读代码的情况下，快速理解当前项目的研究目标、数据来源、处理链路、`Construction State Twin` 逻辑、核心指标、报告生成机制和实验状态。

项目的核心目标不是重新发明一个独立的地质风险预测器，而是围绕“施工报告自动化生成”这一任务，构建一套由数字孪生和大语言模型协同驱动的完整技术框架。

## 2. 项目定位

当前项目最准确的表述不是“风险识别系统”，而是：

**一个面向 TBM 施工报告自动化生成的 Construction State Twin-Driven LLM Framework。**

这里有两个关键界定：

1. 地质风险信息并不是系统从零预测得到的。项目使用的是外部已有地质资料，包括：
   - `TSP`
   - `HSP`
   - `掌子面 / 洞身素描`
2. 大语言模型并不直接读取原始工程文件。系统先构建 `Construction State Twin`，再由 LLM 基于 Twin 生成报告。

## 3. 整体技术路线

项目当前主线可以概括为：

1. 读取 TBM 日运行 `CSV`
2. 解析地质证据 `PDF`
3. 在统一里程轴上做地质-施工融合
4. 计算区段级 `GRS / RAI / GRCI`
5. 构建并递推更新 `Construction State Twin`
6. 基于 `CST` 构造 state-aware prompt
7. 生成正式施工报告与交互式分析结果

这条链的核心思想不是“直接把原始数据喂给模型”，而是“先构建状态，再基于状态写报告”。

## 4. 原始数据来源

### 4.1 TBM 日运行 CSV

主要字段包括：

- 时间字段：`运行时间-time`
- 里程字段：`chainage`、`导向盾首里程`、`刀盘里程`、`主梁里程`、`开累进尺`
- 工况字段：`掘进状态`
- 运行参数：`推力`、`推进速度`、`推进给定速度`、`刀盘扭矩`、`刀盘转速/实际转速`、`贯入度`
- 安全监测：`CO2检测`、`H2S检测`、`SO2检测`、`NO2检测`、`NO检测`、`CH4检测`

### 4.2 地质证据 PDF

当前接入：

- `TSP`：超前地质预报
- `HSP`：超前地质预报
- `掌子面 / 洞身素描`

系统会提取里程范围、围岩等级、灾害标签、出水/掉块/裂隙等结构化信息，并导入证据库。

## 5. 数据处理逻辑

### 5.1 CSV 处理

系统会先完成：

- 时间清洗与排序
- 统一里程轴构建
- 基础工况分段
- 施工状态识别
- 气体统计
- 常规环级停顿候选识别

其中常规环级停顿识别用于在 `RAI` 中减弱正常工序停顿的异常惩罚。

### 5.2 PDF 处理

系统对 `TSP / HSP / 素描` 执行：

- 文本提取
- 格式规范化
- 按文档类型的规则解析
- 证据标准化
- `CSV + SQLite` 双写入库

## 6. 地质-施工融合逻辑

系统先将不同来源的数据统一到里程轴上，然后对每个里程点寻找活跃地质证据，并做事实型融合，而不是直接输出“灾害已发生”。

融合后的关键字段包括：

- `fused_grade`
- `hazard`
- `water_flag_fused`
- `collapse_flag_fused`
- `deformation_flag_fused`
- `active_source_count`
- `weighted_evidence_strength`
- `uncertainty`

随后再按固定区段聚合，形成报告和状态建模的基本对象。

## 7. 三个核心指标

### 7.1 GRS

`GRS` 当前更准确地定义为：

**区段地质关注度表征**

特点：

- 多分量归一化
- 平权聚合
- 高斯衰减平滑
- 与施工响应做有限修正

### 7.2 RAI

`RAI` 定义为：

**区段施工响应异常表征**

特点：

- 以 `Isolation Forest` 为核心
- 输入推力、扭矩、速度、转速、贯入度等参数
- 常规环级停顿候选做惩罚折减

### 7.3 GRCI

`GRCI` 定义为：

**地质关注与施工响应的耦合验证指标**

当前综合：

- 同步性
- 滞后性
- 变化响应
- 多参数一致性

## 8. 当前版本的数字孪生

### 8.1 当前 Twin 的性质

当前版本已经不是单纯的 `digital_twin_state` 快照摘要，而是一个正式的 `recursive CST v2`。

它具备：

- 统一 `CST schema`
- 主链正式产出 `cst_state`
- 状态持久化到 `SQLite`
- 自动查找 `t-1`
- 递推更新 `CST_t`
- 记录变化、稳定性和可信度

### 8.2 当前 Twin 的分层

当前 `CST` 主要包含：

- `temporal_state`
- `spatial_state`
- `operation_state`
- `geological_state`
- `response_state`
- `attention_state`
- `provenance_state`

此外还记录：

- `cst_id`
- `state_key`
- `previous_cst_id`
- `updated_at`
- `llm_summary`
- `raw_twin_state`
- `warnings`
- `gas_state`

### 8.3 当前 Twin 的递推更新

当前已经真实使用上一状态：

\[
CST_t = U(CST_{t-1}, X_t)
\]

其中 `X_t` 包括当前窗口的 PLC、地质证据、掌子面证据和气体状态。

当前 `U` 主要完成：

- 前序状态读取
- 状态差分计算
- `GRS / RAI / GRCI` 变化提取
- `changed_fields` 记录
- `state_confidence` 估计
- `state_stability` 估计
- `persistent_hazards` 更新
- `persistent_attention_segments` 更新
- `trend_label` 生成

它目前属于“结构化递推状态系统”，还不是复杂状态空间模型。

## 9. LLM 协同生成

### 9.1 当前逻辑

当前报告生成已经更接近：

`CST -> State-aware Prompt -> Report`

而不是早期的“直接把分析摘要堆进 prompt”。

### 9.2 约束原则

当前 prompt 主要强调：

- 只依据指定日期或时间窗
- 区分当前掌子面与前方预测
- 风险措辞必须谨慎
- 不允许无证据灾害结论
- 关键结论应能追溯到 `CST`

## 10. Agent 问答链

当前问答系统仍然不是论文主贡献，但它已经是：

**CST 的交互式读取接口**

当前特点：

- Supervisor 架构
- 多角色工具调度
- 会话持久化
- 追问连续性
- 能返回 `cst_state`

所以它更适合作为：

`an interactive extension of CST`

## 11. 当前实验状态

当前项目已经完成了第一轮实验闭环：

- 主对比实验：`Template / Direct-LLM / CST-LLM`
- 主评价指标：`ICS / FCS / TS / RDR / EOR`
- 追溯实验
- 消融实验
- 多源贡献实验
- 轻量级状态连续性分析

这说明当前项目已经不只是“系统搭好了”，而是具备论文实验基础。

## 12. 当前优势

- 已形成统一的状态中间层
- 已完成地质-施工-报告的主链闭环
- 已具备正式 `CST` 递推与落库能力
- 已有可追溯、可约束的报告生成逻辑
- 已有实验脚本、评分表和结果汇总

## 13. 当前仍可继续增强的空间

如果继续往上抬，主要空间在于：

- 增补更好的 `normal case`
- 增补 `w/o indicators`
- 更强人工复核
- 更彻底的 `CST-only` 下游解码
- 更强的动态更新实验
- 更重的状态空间建模

## 14. 一句话总结

当前项目的真正贡献，不是让大语言模型直接读取原始工程数据，而是：

**先将 TBM 施工过程中的多源异构数据构建为一个统一、可递推、可追溯、可被下游任务共同读取的 Construction State Twin，再利用大语言模型基于该状态自动生成正式施工报告。**
